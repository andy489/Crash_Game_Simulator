    Parameter Reference

    Simulation
    ParameterDescriptionValidation
    Spins Total number of individual bets to simulate across all threads. Higher values produce more statistically stable RTP results. 1 – 1,000,000,000
    Thread Count Number of parallel worker threads. Spins are divided evenly across threads (max difference of 1 spin between any two threads). More threads = faster on multi-core machines. 1 – 8
    Stake The fixed amount wagered per spin. Total staked = Stake × Spins. Payout per spin = Stake × multiplier. 0.1 – 200.0, must be a multiple of 0.1
    Zero Spin Interval Every Nth spin is forced to return a 0× multiplier (a dead spin). For example, interval=36 means 1 in 36 spins always pays nothing. Set to 0 to disable. ≥ 0 (0 = disabled)
    Multiplier Range
    ParameterDescriptionValidation
    Min The lowest possible multiplier that a non-zero spin can produce. Acts as the floor of the distribution. Example: 0.1 means the worst live outcome pays back 10% of stake. > 0, multiple of 0.1, < Mid 1
    Mid 1 Boundary between segment 1 and segment 2. Multipliers up to this value are governed by Crash Exponent Low. Beyond it, Crash Exponent Mid takes over. Multiple of 0.1, Min < Mid1 < Mid2
    Mid 2 Boundary between segment 2 and segment 3. Marks where the distribution transitions from the mid steepness to the upper-mid steepness. Multiple of 0.1, Mid1 < Mid2 < Mid3
    Mid 3 Boundary between segment 3 and segment 4 (the high-value tail). Marks where the rare big wins begin. Multiple of 0.1, Mid2 < Mid3 < Max
    Max The absolute maximum multiplier any spin can produce. The distribution is hard-capped at this value. Multiple of 0.1, > Mid3
    Crash Exponents

    Each exponent controls the steepness of the power-law distribution within its segment. Higher values make large multipliers rarer within that segment — the distribution crashes more steeply. Lower values flatten the curve, giving higher multipliers a better chance. The inverse-CDF formula used is: x = boundaryLow / (1 − u)^(1/exponent).

    ParameterSegmentValidation
    Low Min → Mid 1. Controls how quickly the density drops from the minimum. Exponent = 1.0 gives a pure Pareto (power-law) distribution in this segment. > 0
    Mid Mid 1 → Mid 2. Steepness in the medium-value range. Higher values reduce the chance of hitting multipliers above Mid 1. > 0
    Upper Mid Mid 2 → Mid 3. Controls probability of reaching the high-value zone. Typically set much higher than the low/mid exponents to make large wins rare. > 0
    High Mid 3 → Max. Governs the extreme tail — the rarest, highest-paying outcomes. Usually the largest exponent in the config. > 0
    Median Estimation — Knuth Reservoir Sampling

    Computing an exact median requires sorting all values, which would mean storing every spin result in memory. For 500 000 000 spins × 8 bytes = 4 GB per thread — completely impractical. Instead, each thread maintains a fixed-size reservoir of 100 000 randomly-chosen values and estimates the median from that sample. Error stays well under 0.1% regardless of input size.

    Algorithm (per spin)
    1 Increment count (total spins seen so far).
    2 If count ≤ 100 000 — fill the reservoir sequentially. Every early spin is guaranteed a slot.
    3 Otherwise — pick a random integer j in [0, count).
    If j < 100 000, replace reservoir[j] with the new value; otherwise discard it.
    4 At the end, sort the reservoir and return the middle element (or average of the two middle elements).
    Why it works: At any point after the reservoir is full, each of the count values seen so far has exactly a 100 000 / count probability of being in the reservoir — a uniform random sample with no bias toward early or late values. This property was proven by Donald Knuth in The Art of Computer Programming (Algorithm R, Vol. 2).

    Memory: O(k) — fixed at 800 KB per thread regardless of spin count.  |  Time per spin: O(1) amortized — one RNG call and one possible array write.
